# 비용 관점 (cost)

스토리지·로그·아웃바운드 트래픽·메트릭 — 이 스택에서 **비용이 어디서 발생하고, 어떻게 줄이는가**.

> **목적(솔직히)**: 지금은 Supabase 무료 티어 + 로컬 k3s라 **실제 청구액 ≈ 0**이다. 비용 최적화는 본래 *청구서를 측정해 큰 항목을 깎는* 행동인데, 지금은 깎을 비용이 실재하지 않는다. 그래서 이 문서는 **① 비용 드라이버를 식별하고 ② 절감 레버를 정리하되, 그중 상당수는 이미 비용을 의식해 코드에 넣어 둔 것을 "비용 관점"으로 재정리**하고, ③ 아직 안 한 것은 보존주기·샘플링 등 **정책으로 선언**해 둔다. 사용량이 붙을 때 어디를 어떻게 줄일지 미리 정해두는 준비다.

## 비용 드라이버 한눈에

| 드라이버 | 무엇이 비용인가 | 이 스택에서 | 주요 레버 |
|---|---|---|---|
| **스토리지** | 저장 용량(GB·월) | media 버킷의 인증샷 원본·thumbnail | 리사이즈·thumbnail, 중복 제거, orphan 회수, 상한 |
| **로그 적재** | 로그 저장 용량·인덱싱 | 구조화(JSON) 앱/워커 로그 | 레벨, 보존 주기, 고볼륨 이벤트 억제 |
| **아웃바운드 트래픽(egress)** | 외부로 나가는 바이트 | 이미지 조회·grass 임베드 응답 | CDN 캐싱, thumbnail 우선 서빙, 캐시 헤더 |
| **메트릭·트레이싱** | 시계열 카디널리티·스팬 수·보존 | Prometheus 시계열, OTel 트레이스 | 샘플링, 라벨 카디널리티 제한, scrape/보존, 캐시 |

아래에서 각 드라이버를 "무엇이 비용을 키우나 → 어떻게 줄이나(이미 한 것 / 정책)"로 본다.

## 1. 스토리지

**비용을 키우는 것**: 업로드 인증샷이 쌓일수록 선형 증가. 원본 + 파생물(thumbnail) + 참조 잃은 파일(orphan)이 합산된다.

**이미 적용한 레버**
- **thumbnail·메타데이터 후처리** — worker가 업로드 후 이미지를 처리해 `width`/`height`/`checksum`/`thumb_path`를 채운다(`proof_assets`). 목록·미리보기는 원본 대신 작은 thumbnail을 쓰면 저장·전송 둘 다 절감.
- **중복 제거** — `checksum`(sha256) + `proof_assets_checksum_idx`(user 범위)로 동일 파일 재저장을 탐지할 수 있다.
- **orphan 회수** — `/admin/ops`의 orphan object 조회로 `proof_assets`가 참조하지 않는 media 파일을 찾아 정리(미참조 파일이 비용으로 남지 않게). → [운영 콘솔(worklog)](../worklog.md) 참조.
- **상한** — 업로드 8MB 상한을 버킷 정책 + DB constraint(`proof_assets_size_chk`) 양쪽에서 강제해 단일 객체 비용 폭증을 차단.

**정책(선언)**: 원본은 보관하되 목록 서빙은 thumbnail 우선. 실패로 끝난(`failed`/`dead`) asset과 그 원본은 보존 주기 경과 후 정리 대상.

## 2. 로그 적재

**비용을 키우는 것**: 로그량 × 보존 기간 × 인덱싱. 특히 `/health`·`/metrics` 같은 고빈도 폴링 경로가 매 요청 로그를 남기면 양이 폭증한다.

**이미 적용한 레버**
- **레벨 제어** — `LOG_LEVEL`(기본 `info`)로 prod에서 `debug`를 끈다. 구조화(JSON) 로그라 필요한 필드만 남긴다([logging.md](logging.md)).

**정책(선언)**
- **보존 주기** — hot(빠른 조회) 짧게 + cold(아카이브) 길게, 그 뒤 삭제. (아래 표)
- **고볼륨 억제** — health/probe·polling 성공 로그는 샘플링하거나 레벨을 낮춰 양을 줄인다(오류·상태 전이는 전수 유지).

## 3. 아웃바운드 트래픽 (egress)

**비용을 키우는 것**: 이미지 조회가 주 egress. 같은 이미지를 매번 스토리지에서 직접 내보내면 조회수에 비례해 비용이 는다.

**레버(대부분 정책 — 트래픽이 생길 때 적용)**
- **CDN 캐싱** — Storage 앞에 CDN을 두어 반복 조회를 엣지에서 흡수(원본 egress 1회로 수렴). 확장 4단계와 연결([scaling.md](scaling.md)).
- **thumbnail 우선 서빙** — 목록/임베드는 작은 thumbnail로, 원본은 필요 시에만 → 전송 바이트 자체를 줄임.
- **캐시 헤더** — 공개 조회 응답에 `Cache-Control`로 클라이언트·CDN 캐시 적중률을 높여 재전송을 줄인다.

## 4. 메트릭·트레이싱

**비용을 키우는 것**: 시계열 **카디널리티**(라벨 조합 수) × 보존, 그리고 트레이스 **스팬 수**. 고카디널리티 라벨 하나가 시계열을 폭발시킨다. 메트릭 조회 자체도 백엔드 부하(=비용)다.

**이미 적용한 레버**
- **`/metrics` TTL 캐시 + single-flight** — 스크랩이 몰려도 DB 집계를 캐시로 흡수(부하·비용 동시 절감). 부하 진단 회고에서 도입([retrospective/metrics-load.md](../retrospective/metrics-load.md)).
- **집계만 노출** — `metrics_snapshot()`이 상태별 카운트 등 집계만 반환(원본 행 비노출) → 시계열 수를 작게 유지.

**정책(선언)**
- **trace 샘플링** — 전수 수집이 아니라 비율 샘플링(오류·느린 트레이스는 우선)로 스팬 수를 제한.
- **라벨 카디널리티 제한** — `user_id`·`asset_id` 같은 고카디널리티 값을 메트릭 라벨로 쓰지 않는다(로그/트레이스로).
- **scrape interval·보존 주기** — 간격을 너무 짧게 두지 않고, 보존은 아래 표대로.

## 정책 표 (보존 주기 · 샘플링 · thumbnail)

| 항목 | 정책 | 근거 |
|---|---|---|
| **thumbnail** | 업로드 시 worker가 thumbnail 생성, 목록·임베드는 thumbnail 서빙·원본은 on-demand | 스토리지·egress 동시 절감 |
| **이미지 원본** | 8MB 상한(버킷+DB constraint), `checksum` 중복 제거, orphan 주기적 회수 | 단일 객체·중복·미참조 비용 차단 |
| **로그 보존** | hot 단기 + cold 아카이브 후 삭제, prod `LOG_LEVEL=info` | 로그량×기간 절감 |
| **로그 샘플링** | health/probe·polling 성공 로그 억제/샘플, 오류·상태 전이는 전수 | 고볼륨 경로 비용 억제 |
| **trace 샘플링** | 비율 샘플링(오류·slow 우선) | 스팬 수 제한 |
| **메트릭 카디널리티** | 고카디널리티 값 라벨 금지, 집계만 노출, scrape 간격 합리화 | 시계열 폭발 방지 |
| **메트릭 조회** | `/metrics` TTL 캐시+single-flight | 스크랩 부하·비용 흡수 |

## idle 비용 (cross-cutting)

트래픽이 없어도 새는 비용이 둘 있다.
- **컴퓨트 idle** — autoscale을 **기본 off + min replicas 1**로 둬, 부하 없을 때 과금을 최소화([scaling.md](scaling.md)). 켜는 건 실제 트래픽이 생길 때.
- **폴링 부하** — worker가 `jobs`를 주기 폴링(`WORKER_POLL_IDLE_MS`)하므로 트래픽 0에서도 DB 쿼리가 꾸준히 발생한다. idle 간격을 늘리거나, 장기적으로 `LISTEN/NOTIFY` push로 전환하면 idle DB 비용·메트릭이 함께 준다(scaling.md의 DB 완화 단계와 연결).

## 후속

- 실제 청구 환경(클라우드)에서 항목별 실비용을 측정해 이 표의 보존주기·샘플링 수치를 데이터로 확정.
- CDN·로그 보존·trace 샘플링은 정책으로만 선언된 상태 → 트래픽이 생기는 환경에서 적용·검증.
