# 회고 — 비동기 파이프라인(worker·메트릭·trace) 트러블슈팅

DB 큐 + worker 폴링 파이프라인과 그 관측(메트릭·trace)을 붙이며 막힌 지점과 푼 과정을 정리한다. 각 항목은 **증상 → 시도/가설 → 원인 → 해결 → 교훈** 순. (구체 작업 기록은 `worklog.md`, worker 동작은 `architecture/worker.md`, 운영 절차는 `runbooks/runbook.md`.)

## 1. 빈 큐에서 worker가 "유령 job"을 처리

- **증상**: 큐가 비었는데도 worker가 매 폴링마다 job을 잡아 처리하려는 것처럼 보였다.
- **시도/가설**: 폴링 로직을 의심하다가, `claim_job`의 반환값을 직접 들여다봄.
- **원인**: `claim_job(RETURNS public.jobs)`은 빈 큐에서 NULL을 반환하는데, PostgREST가 이를 **"전 컬럼이 null인 한 행"** 으로 표현한다. worker는 "행이 왔다"만 보고 처리하러 들어갔다.
- **해결**: 반환 행의 **`job.id == null` 가드**를 두어 실제 선점 여부를 판별. id가 없으면 빈 큐로 보고 백오프.
- **교훈**: 함수가 "값이 없음"을 표현하는 방식(NULL)과 그것을 실어 나르는 계층(PostgREST)의 표현이 다를 수 있다. "행이 있나"가 아니라 **의미 있는 키(id)가 있나**로 판별해야 한다.

## 2. 잘린 service 키로 `Invalid API key` 무한 재시도

- **증상**: 실패 처리(재시도/백오프)를 검증하던 중, worker가 `claim_job`에서 `Invalid API key`로 계속 실패. 그런데 worker는 죽지 않고 **폴링을 무한 재시도**해 설정 오류가 가려졌다.
- **시도/가설**: 처음엔 키 자체가 만료/오권한인 줄 의심. 실값을 출력하지 않고 키의 **형식**만 점검.
- **원인**: `.env.local`의 service_role 키가 복붙 과정에서 **잘려(truncated)** 있었다. JWT는 길이 ~200·점(`.`) 2개 구조인데, 잘린 값은 그 형태를 깨뜨린다.
- **해결**: 키 **길이·점 개수**로 진단(실값 노출 없이) → 온전한 키로 교체. 이 함정을 runbook의 incident 항목으로 정식 기록.
- **교훈**: worker의 복원력(죽지 않고 재시도)이 **설정 오류를 숨긴다** — 장애처럼 안 보이고 조용히 안 도는 상태. 시크릿은 실값을 보지 않고 **형식(길이·구분자 개수)** 으로 검증할 수 있다. "복원력 있는 컴포넌트일수록 설정 오류를 드러내는 진단 로그가 필요하다."

## 3. 처리 지연 지표가 22997초로 폭발

- **증상**: `/metrics`에 추가한 처리 지연 평균이 ~22997초로 비현실적으로 컸다.
- **시도/가설**: worker가 실제로 한 건에 몇 초 걸리는지(로그상 ~2초)와 지표가 안 맞는 점에 주목 → 무엇을 빼고 무엇을 더해 "지연"을 정의했는지 되짚음.
- **원인**: 평균을 `created_at→updated_at`(enqueue→done)으로 쟀다. dev에서 worker를 띄엄띄엄 돌리니 **큐 대기 시간(수 시간)** 이 그 구간에 섞여, "처리 시간"이 아니라 "접수~완료 전체"가 됐다.
- **해결**: 기준을 **`locked_at`(claim)→`updated_at`(done)** 으로 바꿔 **큐 대기를 제외한 순수 처리 시간**(~1.7초, worker 로그와 일치)을 재게 함.
- **교훈**: 같은 두 타임스탬프라도 **무엇을 측정하려는가**에 따라 시작점이 다르다. "처리 지연"은 처리 시작(claim)부터다. 지표가 폭발하면 값보다 **정의(어느 구간을 재는가)** 를 먼저 의심.

## 4. trace_id가 null로 보임 (구 asset/배포 혼입)

- **증상**: web→worker 추적용 `trace_id`를 붙였는데, worker 로그에서 일부 자산의 `trace_id`가 계속 `null`로 보였다.
- **시도/가설**: 전파 코드 버그를 의심했으나, 같은 worker 실행에서 **어떤 자산은 채워지고 어떤 자산은 null** 인 점이 단서.
- **원인**: null인 자산은 `trace_id`를 박는 코드가 없던 **구 코드(Vercel/main에 배포된 버전)** 로 올라온 옛 자산이었다. 새 코드로 올린 자산만 trace_id가 있었다.
- **해결**: trace_id를 부여하는 코드가 도는 환경에서(로컬 dev 업로드) 새 자산을 올려 검증 → 새 자산은 trace_id가 채워지고, 옛 자산은 null로 대비됨(before/after가 한 화면에 드러남).
- **교훈**: "일부만 null"은 코드 버그가 아니라 **데이터 출처(어느 버전이 만든 행인가)** 의 차이일 수 있다. 비동기 파이프라인은 옛 데이터가 큐/DB에 남아 섞이므로, 검증은 **새 입력으로** 하고 옛 데이터와의 대비를 증거로 쓴다.

---

## 5. dev 서버가 옛 라우트/에셋을 서빙

- **증상**: 라우트/메트릭을 바꿨는데도 브라우저·dev 터미널이 옛 동작을 보임.
- **원인**: dev 서버가 옛 빌드/캐시를 서빙하거나 브라우저가 옛 에셋을 캐시.
- **해결**: dev 서버 **재시작 + 하드 리프레시**(캐시 비활성). 이후 의도한 라우트/구조화 로그가 정상 노출.
- **교훈**: "코드는 고쳤는데 동작이 안 바뀜"이면 **서빙 계층(dev 서버·브라우저 캐시)** 을 먼저 의심하고 재시작·하드 리프레시로 변수부터 제거한다. (관측 작업에서 특히, 옛 출력에 속아 엉뚱한 곳을 파지 않게.)

---

## 전체적으로 배운 것

- 비동기 파이프라인의 함정은 **"없음/지연/옛 데이터"의 표현**에 몰려 있다 — NULL의 표현(유령 job), 큐 대기 혼입(지연 폭발), 옛 버전 데이터(trace_id null).
- worker의 **복원력은 양날**이다 — 죽지 않아 좋지만 설정 오류를 숨긴다. 진단 로그와 형식 검증으로 보완.
- 지표·trace는 **정의와 출처**를 먼저 의심한다(값이 이상하면 측정 구간·데이터 버전부터).
- 시크릿은 **실값을 보지 않고** 형식으로 진단한다.
