# 회고: API 입력 검증과 신뢰 경계 (input validation & trust boundary)

권한 경계 감사 중 `/api/proof-assets`에서 **클라이언트가 보낸 `source_path`를 그대로 등록**하는 걸 발견했다. "로그인은 했으니 괜찮겠지"가 아니라, 인증된 사용자라도 *보낸 값*은 따로 검증해야 한다는 걸 다시 새긴 작업. 그 과정에서 정리한 개념을 남긴다.

> 결과: source_path 소유 검증(403) + zod로 content_type/size/kind 검증(400) 추가. 절차·증거는 [worklog](../worklog.md), 여기선 *왜 그래야 하는가*.

## 신뢰 경계 (trust boundary)

네트워크 경계를 넘어 들어온 입력은 **검증되기 전까지 적대적이라고 가정**한다. 브라우저의 우리 JS(`upload.ts`)가 검증을 하더라도, 그건 **공격자가 우회할 수 있는 쪽**(클라이언트)에 있다 — DevTools 콘솔에서 `fetch` 한 줄이면 우리 검증 코드를 건너뛰고 서버로 임의의 바디를 보낼 수 있다. 그래서 **진짜 게이트는 항상 서버**여야 한다.

핵심 한 줄: **클라이언트 검증은 UX(빠른 피드백)용, 보안 게이트가 아니다.**

## 왜 TypeScript로는 안 되나

흔한 착각: "타입을 `{ source_path: string }`으로 적었으니 런타임에도 string이겠지." 아니다.
- **TypeScript 타입은 컴파일 시점에만 존재**하고, 빌드되면 사라진다.
- 런타임의 `await req.json()`은 그냥 `any`/`unknown` — 클라이언트가 숫자를 넣든 객체를 넣든 타입이 막아주지 않는다.

→ 그래서 **런타임 검증**(수동 `if` 또는 zod 같은 스키마)이 *별도로* 필요하다. 타입은 검증이 아니다.

## 이 앱에서 진짜 위험했던 이유

`source_path` 무검증이 단순한 "지저분함"이 아니라 **실제 취약점**이었던 경로:

1. 공격자가 로그인한 상태로 `POST /api/proof-assets` 에 `source_path = "<피해자_uid>/doits/x.jpg"` 를 보낸다.
2. 서버는 그걸 그대로 `proof_assets`에 등록(자기 `user_id`로). proof_assets RLS는 `user_id`만 보지 `source_path` 내용은 안 본다.
3. 등록되면 트리거가 `jobs`를 enqueue → **worker가 그 job을 집는다.**
4. worker는 **`service_role`로 접속해 RLS를 우회**하므로, 피해자 폴더의 파일을 **다운로드·처리**한다(썸네일·메타데이터 등).

즉 **다른 사용자의 파일을 내 자산으로 끌어오는 IDOR성 노출**. RLS가 곳곳에 있어도, *경계 하나(API 등록)가 검증을 빼먹으면* 뒤의 service_role 권한과 연결돼 도미노가 된다.

교훈: **권한 경계는 "한 군데만 뚫려도" 무너진다.** 특히 뒤에 service_role처럼 강한 권한이 있으면, 그 앞단 입력 검증이 곧 방어선이다.

## manual 검증 vs zod

처음엔 손으로 `if (typeof x !== "string" || ...)`로 막았고(동작·보안 동일), 이후 **zod 스키마**로 바꿨다.
- **보안 효과는 같다** — zod가 새 보안을 더해주는 게 아니다.
- zod의 이점: **선언형**(스키마만 보면 계약이 보임), 일관된 에러, 타입 추론, **라우트가 늘수록** 유리. 현업 표준.
- 비용: 의존성 1개. (이 프로젝트는 drvfs라 native 빌드 패키지 설치가 막히지만, zod는 순수 JS라 로컬·컨테이너 모두 설치됨.)

선택: 라우트가 하나뿐인 지금은 manual로도 충분했지만, "제대로 된 방식"을 보이려 **zod(B안)** 채택.

### 테스트로 관측되는 차이 (manual vs zod)

같은 콘솔 테스트(`fetch`로 잘못된 바디 전송)를 두 버전에 돌렸을 때, **출력에서 실제로 보이는 차이**:

| | manual 검증 | zod 검증 |
|---|---|---|
| 거부 응답 본문 | (상태코드만, 예: `400`) | `{ error, details: { 필드: [메시지] } }` — **필드별** 사유 |
| 다중 위반 | 순차 `if`라 **첫 실패에서 return** → 하나만 | 스키마 전체 검사 → **틀린 필드를 모두 모아** 반환 |
| 증거 | `147`(상태코드 `400`만) | `148`(`details:{content_type, size_bytes}` — 두 위반 동시) |

- 처음 테스트(자료 `146`/`147`)는 `console.log(r.status)`로 **상태코드만** 찍어, manual이든 zod든 화면이 동일했다("거부됨"만 보임).
- zod 검증 후 테스트(자료 `148`/`149`)는 `console.log(r.status, await r.json())`로 **본문까지** 찍어, `details`에 **content_type·size_bytes 두 위반이 한 번에** 잡힌 게 보인다 — 이게 manual 순차검증과 구별되는 zod의 관측 가능한 동작이다.

교훈: 검증의 *효과*는 상태코드로 보지만, **검증의 "품질"(무엇을·얼마나 잡는지)은 응답 본문을 봐야 드러난다.** 테스트도 상태코드만 보지 말고 본문까지 확인할 것.

## 403 vs 400 — 왜 나눴나

검증 실패를 한 덩어리로 400 처리하지 않고 구분했다:
- **403 Forbidden** — `source_path`가 **본인 소유가 아님**(authorization 위반). "요청 형식은 맞지만 그럴 권한이 없다." `user.id`가 필요해 인증 뒤 런타임에서 체크.
- **400 Bad Request** — content_type/size/kind 등 **형식·범위가 틀림**(shape 위반). zod가 선검증.

의미를 코드(상태 코드)로 드러내면 로그·모니터링에서 "권한 공격 시도(403)"와 "잘못된 요청(400)"을 구분할 수 있다.

## 교훈 정리

- **클라이언트 검증은 UX용, 서버 검증이 보안 게이트.** 우회 가능한 쪽을 믿지 않는다.
- **타입 ≠ 검증.** TS 타입은 런타임에 사라진다 → 런타임 스키마 검증 필요.
- **단일 게이트 금물.** 버킷(MIME·size) + API guard(소유·shape)처럼 이중으로. 한 경계의 누락이 뒤의 강한 권한(service_role)과 연결되면 도미노.
- **상태 코드로 의미 표현**(403=authorization, 400=shape) → 운영 가시성.
