# 회고 — `/metrics` 부하 시 ~10초 행: "느린 곳"을 3-way로 가른 과정

k6로 성능 베이스라인을 재다가 `/metrics`(DB 읽기 경로)가 부하에서 100% 실패했다. "DB가 느린가?"로 단정하지 않고 어디가 느린지 좁혀 간 기록.

- **증상**: `health_live`(순수 앱)는 296 rps·에러 0%인데, `metrics_read`(`/metrics`, `metrics_snapshot` RPC)는 **20초 동안 40건만 처리, 전부 ~10.5초 타임아웃·100% 실패**. 단건 호출은 정상(배포 후 smoke 통과).

- **가설1 → 반증**: "동시성으로 DB 커넥션 포화." → **캐시 + single-flight**(TTL 동안 DB 미접근, 동시 요청은 한 번의 조회 공유)를 넣었더니 **좋은 구간에선 8천 건 처리**(개선은 진짜). 그런데 **저부하(VUS=5)에서 오히려 100% 실패** → 동시성이 원인이 아님이 드러남.

- **3-way 진단(핵심)**: "느린 곳"을 경로별로 격리.
  | 경로 | 시간 | 판정 |
  |---|---|---|
  | 쿼리 직접 `explain analyze select metrics_snapshot()` | **3.6 ms** | 쿼리는 안 무겁다 |
  | REST API: 개발자 머신 → PostgREST `curl` ×3 | **~130 ms** | Supabase API도 멀쩡 |
  | 앱 pod 경유(부하) | **~10 s, 100% 실패** | 여기만 느리다 |
  → 쿼리·API 다 빠른데 **앱 pod에서 나가는 호출만** 느림.

- **원인**: **pod ↔ Supabase 네트워크 경로가 시간에 따라 간헐적으로 죽는 환경성 문제**(로컬 WSL2/k3s → 인터넷 추정 — pod의 stale keep-alive 연결이 ~10초 소켓 타임아웃). 부하 무관하게 좋은/나쁜 구간이 분 단위로 바뀜(VUS=20 반복: 통과 ↔ 70% ↔ 100% 실패). 갓 뜬 pod는 빠르다가 시간이 지나며 굳음.

- **해결(앱이 할 수 있는 것)**: rpc에 **AbortController fail-fast 타임아웃(2s)** → 행을 10초→2초로 끊고 그 소켓을 버려 다음 호출은 새 연결로 자가복구 유도. + **serve-stale**(신선화 실패 시 직전 정상값 반환) → 나쁜 구간에도 200으로 생존. 좋은 구간 p95 112ms/0%, 나쁜 구간 2s fail-fast + 부분 생존.

- **교훈**:
  - **"느리다 → DB 탓"으로 점프하지 말 것.** 쿼리/게이트웨이/앱-경로를 **각각 따로 재면** 원인이 한 곳으로 좁혀진다(여기선 3.6ms / 130ms / 10s로 깔끔히 갈림).
  - **개선이 한 번에 다 풀리지 않는다.** 캐시는 동시성을 풀었지만 진짜 병목(연결)은 따로였다 — 반증이 나오면 가설을 버리고 다시 측정.
  - **환경성 한계는 앱 코드로 "가린다"(fail-fast·stale)지 "없애지" 못한다.** 정직하게 "이건 로컬 클러스터 네트워크 한계"라고 적고, 진짜 검증은 안정적 네트워크의 클라우드 클러스터로 미룬다.
