# 회고: HSTS와 preload — 되돌리기 비용이 다른 두 가지

Security headers를 넣으며 HSTS에 `preload`를 붙일지 따져봤다. 보안은 강해지지만 **되돌리기 비용이 일반 HSTS와 차원이 다르다**는 걸 알게 돼, 지금은 빼고 가기로 한 기록.

> 결론: 헤더 값은 `Strict-Transport-Security: max-age=63072000; includeSubDomains` — **`preload` 제외**. 영구 HTTPS 확신이 서면 후속으로 등록.

## HSTS란

`Strict-Transport-Security` 응답 헤더는 브라우저에게 "**이 도메인은 `max-age` 동안 HTTPS로만 접속하라**"고 지시한다. 한 번 받으면 브라우저가 기억해서, 사용자가 `http://`로 가도 자동으로 `https://`로 바꾼다(다운그레이드·중간자 공격 차단).

- `includeSubDomains` — 모든 서브도메인에도 적용.
- `preload` — 아래의 특별한 의미.

## 일반 HSTS는 되돌릴 수 있다

`preload` 없이 헤더만 보내는 경우, HSTS 상태는 **각 방문자 브라우저가 내 서버 응답을 받아서** 기억하는 것이다. 그래서 끄려면 **`max-age=0`을 응답으로 내려보내면** 그 브라우저가 잊는다. 내 서버가 통제권을 쥐고 있다 — 헤더로 켜고, 헤더로 끈다.

## preload는 "남의 목록에 박히는" 것이라 되돌리기 어렵다

`preload`를 붙이고 **hstspreload.org에 도메인을 등록**하면, 그 도메인이 **Chrome·Firefox·Safari 등 브라우저에 하드코딩된 목록(preload list)으로 내장**된다. 차이가 여기서 갈린다:

1. **내 서버를 거치지 않고도** 브라우저가 "이 도메인은 무조건 HTTPS"라고 안다 — 첫 방문, 헤더를 못 보낸 상황에서도. (그게 preload의 장점: 최초 접속의 빈틈까지 막음.)
2. 그래서 **끄려면** preload 제거를 신청하고 → 브라우저 목록에서 빠진 게 → **브라우저 업데이트로 사용자에게 배포될 때까지** 기다려야 한다 → **수주~수개월**. 내 손을 떠나 있다.
3. 그동안 `includeSubDomains`까지 박혀 있으면 **모든 서브도메인이 HTTPS 강제** — 어떤 서브도메인이 HTTPS를 못 하면 **접속 자체가 막힌다**.

즉 일반 HSTS는 "내가 헤더로 끌 수 있음", preload는 "**브라우저 벤더 목록에 박혀 내 맘대로 못 뺌**". 같은 단어에 묶여 있지만 되돌리기 비용이 전혀 다르다.

## 그래서 내린 선택

- 지금은 **`preload` 제외**. `max-age` + `includeSubDomains`만으로 보안 효과의 대부분을 얻으면서, 되돌리기 가능한 상태를 유지한다.
- preload 등록은 **도메인과 모든 서브도메인이 영구히 HTTPS라고 확신**이 설 때 후속으로. (보안 등급 평가에 preload가 필수도 아니다.)

## 교훈

- **"보안을 더 강하게"와 "되돌릴 수 있나"는 따로 본다.** 더 강한 설정이 항상 옳은 게 아니라, 잘못됐을 때 빠져나올 수 있는지가 운영에선 똑같이 중요하다.
- **통제권이 내 손에 있는지** 따져라 — 헤더(내 서버) vs 브라우저 내장 목록(벤더). 후자는 한 번 들어가면 내 배포로 못 되돌린다.
- 같은 기능 이름 안에도 **롤백 비용이 다른 옵션**이 섞여 있을 수 있다(HSTS는 끄기 쉬움 / +preload는 끄기 어려움). 옵션 단위로 되돌리기 비용을 본다.
