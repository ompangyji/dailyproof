# 회고 — 테스트 계층: E2E는 왜, 어디에 넣는가

E2E 테스트를 추가하면서 "이건 왜 하는 거지?"가 바로 와닿지 않았다. 이미 smoke·health check·유닛 테스트가 있는데 E2E가 무엇을 더 해주는지, 어디서 도는지를 정리한다.

## 이전 상태 — 무엇이 있었고 무엇이 비어 있었나

테스트성 장치는 이미 여러 개 있었지만, 각자 **다른 것**을 본다.

| 장치 | 무엇을 검증 | 한계 |
|------|------------|------|
| 유닛 테스트 (`node --test`, `lib/**`) | 함수 하나의 입출력 | 함수는 맞아도 화면·라우팅이 맞는지는 모름 |
| health check (`/health/live`·`/ready`) | 서버가 살아있나 / 의존성 도달 | "200을 준다"일 뿐, 사용자 흐름은 모름 |
| smoke (`scripts/smoke.mjs`) | 배포 직후 health·metrics를 curl | HTTP 레벨 점검, 브라우저 동작은 모름 |

→ **빠진 층**: "브라우저로 실제 화면을 열었을 때, 사용자 흐름이 의도대로 도는가." 계획서의 "테스트 자동화" 완료 기준에도 **최소 E2E 1개**가 명시돼 있는데 그것만 비어 있었다.

## E2E가 채우는 자리 — 테스트 피라미드

- **유닛**: 작은 부품(함수)이 맞나. 빠르고 많이.
- **통합/HTTP(health·smoke)**: 부품들이 붙어 서버로서 응답하나.
- **E2E**: 사용자가 실제로 겪는 흐름(브라우저 → 라우팅 → 렌더 → 상호작용)이 끝까지 동작하나. 느리지만 "진짜 사용자 관점"을 본다.

세 층은 대체재가 아니라 **보완재**다. 유닛이 다 통과해도 "비로그인인데 보호 페이지가 그냥 열려버리는" 회귀는 유닛으로 못 잡는다 — 그건 미들웨어·라우팅·렌더가 엮인 흐름이라 E2E의 몫이다.

## 무엇을 E2E 대상으로 골랐나 — 그리고 왜

**대상**: 비로그인 사용자가 보호 경로(`/`)에 가면 `/login`으로 리다이렉트되고 로그인 폼이 떠야 한다(+ Sign up 링크 이동).

고른 이유:
- **결정적(deterministic)**: 실제 Supabase 계정·시크릿 없이, 더미 공개키만으로 항상 같은 결과가 나온다. 로그인 호출 대상을 미기동 포트로 두면 supabase 호출이 즉시 실패 → 미들웨어가 '비로그인'으로 처리 → 리다이렉트. 외부 상태에 흔들리지 않아 CI에서 안정적이다.
- **가치 있는 동작**: "비로그인은 못 들어온다"는 **보안상 핵심 가드**다. 회귀하면 치명적이라 검증 가치가 높다.
- 반대로 "로그인 후 대시보드"는 테스트 계정·실데이터·시크릿이 필요해 CI에서 **비결정적**이 되기 쉬워 최소 E2E로는 일부러 피했다.

## 어디서 도는가 — CI merge 게이트의 한 겹

- 로컬 실행도 가능하지만(`npm run build` → `npm run test:e2e`), 핵심은 **CI에 `e2e` 잡으로 묶은 것**: PR마다 빌드 → `next start` 기동 → Playwright로 흐름 검증 → 실패 시 merge 차단.
- 이는 배포 후 자동 smoke 게이트(ArgoCD PostSync hook, `cicd-gitops.md` 9절)의 **"배포 전(merge 전)" 버전**이다. 같은 "자동으로 비정상을 잡아 막는다"를 **다른 시점**에 건 것:
  - **CI E2E** = merge 전, 코드가 사용자 흐름을 깨뜨렸으면 막는다.
  - **PostSync smoke** = 배포 후, 실제 환경에서 깨졌으면 sync를 막는다.

## 교훈

- "테스트가 있다"가 아니라 **"어느 층을 보느냐"** 가 중요하다. smoke·health·유닛이 있어도 E2E가 보는 사용자 흐름은 별개다.
- E2E는 **결정적으로 만들 수 있는 흐름**부터 고른다(외부 의존·시크릿 최소화). 한 개라도 핵심 가드를 덮으면 회귀 안전망이 생긴다.
- 자동 검증은 **시점별로** 건다 — merge 전(CI)과 배포 후(PostSync). 둘은 겹치는 게 아니라 서로 다른 구간을 막는다.
