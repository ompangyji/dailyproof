## 1. 거대 worklog 문서에서 Notion API 타임아웃

- **증상**: `worklog.md`(약 738블록)처럼 큰 문서를 동기화할 때 sync가 Notion API 타임아웃으로 실패. 작은 문서는 정상.
- **시도/가설**: 문서 크기에 비례해 실패하는 점에서 "본문 교체 과정의 왕복 횟수"를 의심.
- **원인**: 본문을 교체하려면 기존 블록을 먼저 지워야 하는데, `clearPage`가 블록을 **하나씩 순차 삭제**했다. 수백 개 블록이면 수백 번의 API 왕복이 직렬로 일어나, 클라이언트 기본 타임아웃(60s)을 넘겼다.
- **해결**:
  - `clearPage`를 **동시성 제한(8) 병렬 삭제**로 전환(페이지 id는 유지, 본문 블록만 제거).
  - 모든 Notion 호출을 `withRetry`로 감쌈 — 타임아웃/429/5xx에 지수 백오프(1→2→4s, 4회).
  - 클라이언트 `timeoutMs`를 120s로 상향. append는 **순서 보존**을 위해 순차로 두되 재시도만 적용.
- **교훈**: "크면 실패"는 보통 **N번의 직렬 왕복**이 원인이다 — 순차를 동시성 제한 병렬로 바꾸면 단번에 풀린다(무제한 병렬은 rate limit을 부르니 상한을 둔다). 순서가 중요한 작업(append)과 안 중요한 작업(delete)을 구분해, 삭제만 병렬화하고 추가는 순차로 둔 게 핵심. 외부 API는 재시도(백오프)를 기본값으로 깐다.

## 전체적으로 배운 것

- 외부 API 동기화에서 실패의 1차 용의자는 **호출 횟수×직렬성**이다. 병렬화(상한 포함) + 재시도 + 타임아웃 상향이 표준 처방.
- **순서 의존성**을 기준으로 작업을 나눈다 — 순서 무관(delete)은 병렬, 순서 보존 필요(append)는 순차.
- 헬퍼는 단위테스트로 고정(순서보존·동시성≤8·재시도·즉시 throw·소진 throw)하고, 실제 API 검증은 다음 docs push의 CI 재실행으로 확인 — **순수 로직과 통합 검증을 분리**.
