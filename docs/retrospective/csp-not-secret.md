# 회고: CSP는 숨기는 게 아니라 강제하는 방어다

Security headers를 넣다가 든 의문: "CSP가 켜진 걸 공격자가 알면(헤더·콘솔 위반 로그로) 보안 세팅을 했다는 힌트를 주는 것 아닌가?" 답을 정리하면서 **"방어를 안다 ≠ 방어를 뚫는다"** 라는 보안의 기본을 다시 새긴 기록.

## 의문

CSP 위반이 나면 콘솔에 `Refused to ... because it violates ... Content Security Policy`가 찍힌다. 악의적 요청으로 이런 로그를 유발하면, **"이 사이트는 CSP로 막고 있다"는 정보를 노출**하는 것 아닌가? 그게 공격자에게 힌트가 되지 않나?

## 정리 1 — CSP는 원래 공개된다

CSP는 **응답 헤더로 브라우저에 내려보내는** 정책이다. 숨겨질 수가 없다.
- F12 → Network → 응답 헤더에 `Content-Security-Policy` 전문이 그대로 보인다.
- `curl -I https://도메인` 한 줄이면 디렉티브 전체가 나온다.

즉 공격자는 콘솔 위반 로그를 *유발하지 않아도* 헤더만 보면 정책을 다 안다. 콘솔 로그는 추가 힌트조차 아니다. **CSP가 켜진 것을 아는 것은 막을 수 있는 일이 아니고, 막을 필요도 없다.**

## 정리 2 — 숨김(obscurity)이 아니라 강제(enforcement)

보안 통제는 성격이 둘로 갈린다.
- **security by obscurity** — 존재를 모르게 해서 지킴. 알려지면 무력화된다. (약한 의존)
- **enforcement** — 알려져도 못 뚫는다. CSP가 이쪽이다.

CSP는 "self·이 nonce가 붙은 스크립트만 실행"을 **브라우저가 강제**한다. 공격자가 "script-src에 nonce를 쓰는구나"를 **알아도**, 그 nonce는 **요청마다 바뀌는 난수**라 예측·재사용이 안 된다. 정책을 안다고 우회가 쉬워지지 않는다.

이게 **케르크호프스 원칙(Kerckhoffs's principle)** 이다: *시스템의 구조가 전부 공개돼도, 비밀(여기선 요청별 nonce)만 지켜지면 안전해야 한다.* 좋은 보안은 "비밀 = 시스템 설계"가 아니라 "비밀 = 키/난수"에 의존한다.

## 정리 3 — 오히려 약한 CSP일수록 공개가 아프다

- **강한 CSP**(nonce·strict-dynamic·인라인 차단)는 공개돼도 칠 게 없다. 알아봤자 우회 수단이 안 보인다.
- **느슨한 CSP**(`script-src 'unsafe-inline'` 등)는 공개돼 있으니 "인라인 허용하네 → XSS 여지"가 드러난다.

→ 그래서 우리가 **nonce 방식(강한 CSP)** 을 고른 게 의미가 있다. "알려져도 안전한 쪽"을 택한 것.

## 그럼 진짜로 줄여야 할 노출은 — 기술스택 힌트

CSP는 숨기지 않지만(숨기면 강제가 작동 안 함), **굳이 줄 필요 없는 정보**는 따로 있다.
- `Server`, `X-Powered-By` 같은 **기술스택 노출 헤더** — 공격자가 "Next.js구나 → 그 버전 알려진 취약점 노려볼까"의 출발점이 된다.
- 에러 페이지의 스택트레이스·내부 경로.

이건 방어(enforcement)가 아니라 **공격 표면 정찰을 덜 돕는** 정보 위생이다. 그래서 이번에 `next.config`에 **`poweredByHeader: false`** 를 넣어 `X-Powered-By: Next.js`를 제거했다.

## 교훈

- **CSP가 켜진 걸 아는 것은 힌트가 아니다.** 헤더로 원래 공개되며, 강제형 통제라 알아도 못 뚫는다(nonce는 난수).
- **숨김에 의존하지 말고 강제에 의존하라**(케르크호프스). 비밀은 설계가 아니라 키/난수여야 한다.
- **단, "강제 못 하는 정보"는 줄인다** — 기술스택 헤더(`X-Powered-By`) 제거 같은 정보 위생은 별개로 한다. CSP를 숨기는 것과 혼동하지 않는다.
