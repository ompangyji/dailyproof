# rate limiting — 대상 선정 분석과 정책

rate limit(일정 시간 안에 요청이 N회를 넘으면 `429 Too Many Requests`로 거부)을 **어느 엔드포인트에, 어떤 한도로, 왜** 거는지 따진 기록. 결론만 코드에 넣지 않고 *근거*를 남긴다.

> **목적(솔직히)**: 지금은 실트래픽이 없어 "공격받아서"가 아니다. 위험·비용은 **측정값이 아니라 추론**(threat-model식 가정)이다. 이 문서는 ① 어디가 노출·비용이 큰지 기준을 세워 평가하고 ② 그 근거로 대상·한도를 정하며 ③ 앱 레벨 최소 구현으로 실증하되 한계(진짜 운영은 edge/Redis)를 명시한다.

## 평가 기준 (무엇으로 위험·비용을 봤나)

| 기준 | 의미 |
|---|---|
| **노출도** | 비로그인 공개인가, 인증 게이트 뒤인가 (공개일수록 외부 공격에 직접 노출) |
| **악용 시나리오** | DoS, 토큰 brute-force, scraping, 스팸 기록 등 구체적 남용 방식 |
| **영향(비용)** | 남용 시 무엇이 비싸지나 — DB 쓰기·큐 적체·egress 대역폭 |
| **기존 방어** | 이미 막는 장치(인증·RLS·캐시 등)가 있어 rate limit 필요도가 낮아지나 |

## 엔드포인트별 평가

| 라우트 | 노출도 | 악용 시나리오 | 영향(비용) | 기존 방어 | 우선도 |
|---|---|---|---|---|---|
| `/api/grass/[token]` | **공개·비로그인** | 토큰 brute-force(`[a-f0-9]{8,64}`), scraping, DoS | DB read(get_grass) — 단 응답 캐시됨 | 토큰 추측 난이도 + `Cache-Control` 600s | **1 (최상)** |
| `/api/proof-assets` (POST) | 인증 필요 | 스팸 등록 → `jobs` 큐·DB 적체 | DB write + job enqueue | 인증 게이트 + API guard(소유·shape) | **2** |
| `/api/media/[...path]` | 인증 필요 | 대량 다운로드 | egress 대역폭 | 인증 + RLS read-own + `Cache-Control` | 3 |
| `/login`·`/signup` (Supabase auth) | 공개 | 비밀번호 무차별 대입 | 인증 시스템 부하 | **Supabase Auth 자체 rate limit** 내장 | 위임(우리 X) |
| `/api/health`·`/metrics` | 공개/probe | 폴링 남용 | 낮음 | `/metrics` TTL 캐시 | 낮음 |

## 결론 — 어디에, 어떤 한도로

1. **`/api/grass/[token]` — 1순위.** 유일한 **비로그인 공개** 엔드포인트라 외부 공격에 그대로 노출된다(다른 곳은 인증이 1차 방어선). 토큰 brute-force·scraping을 늦추려면 **IP 기준** rate limit이 가장 정당하다.
   - 한도(초안): **IP당 분당 60회**. 정상 임베드(브라우저가 캐시된 이미지를 가끔 갱신)엔 충분, 자동화 남용엔 제동.
2. **`/api/proof-assets` (POST) — 2순위.** 인증 뒤지만 **DB 쓰기 + job 생성**이라 스팸 등록이 큐·DB에 부담. **사용자(uid) 기준** 한도.
   - 한도(초안): **uid당 분당 30회.** 정상 업로드 빈도를 크게 상회.
3. **`/api/media` — 보류(3순위).** 인증+RLS+캐시로 1차 방어가 있고, 진짜 egress 보호는 CDN 단계가 적합([scaling.md]·[cost.md]) → 지금은 제외, 후속.
4. **로그인/회원가입 — 위임.** Supabase Auth가 자체 rate limit을 가지므로 앱에서 중복 구현하지 않는다(중복은 오히려 정상 사용자 차단 위험).

## 구현 방식과 한계

- **앱 레벨 in-memory** rate limiter: `(IP 또는 uid)+라우트`를 키로 고정 윈도우 카운트, 초과 시 **429 + `Retry-After`** 반환.
- **한계(반드시 명시)**:
  - **다중 pod에선 per-pod** — 인스턴스마다 카운터가 따로라 전체 한도가 pod 수만큼 느슨해진다.
  - **재시작 시 초기화** — 메모리라 영속성 없음.
  - → **진짜 운영 해법**은 ① **edge(Traefik Middleware / Ingress)** 에서 걸거나 ② **공유 저장소(Redis)** 기반 분산 rate limit. 이 앱은 단일 노드 데모라 in-memory로 *통제의 존재·동작*을 실증하고, 분산은 정책으로 선언.

## 검증 계획

- 대상 라우트를 한도 이상으로 반복 호출 → **429** 응답 + `Retry-After` 확인(스샷). 한도 이하에선 정상 통과.

## 후속

- 실트래픽 환경에서 정상 사용 패턴을 측정해 한도 수치를 데이터로 보정.
- 분산 rate limit(edge/Redis) 적용 — 다중 pod 환경에서 필수.
